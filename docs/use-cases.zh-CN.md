# 使用场景

`ai-approval-workflow` 适合定时自动化任务：产出简短 AI 摘要，并且只在需要简单决策时请求人工输入。

> 所有示例都使用通用服务名、占位路径和公开安全的描述。内部主机名、凭证和命令应放在部署专属配置中。

## 1. 每日 GitHub Trending 摘要

**问题：** 手动跟踪新库和新工具会产生重复浏览成本。

**工作流形态：**

```text
每天 09:00
  -> 抓取 GitHub Trending
  -> AI 总结值得关注的仓库、语言分布和想法
  -> 发送通知
```

**结果：** 用简短日报替代手动浏览。该任务只读，不需要审批。

**示例文件：** `examples/github-trending-daily.yaml`

## 2. 服务升级检查 + 审批

**问题：** 服务发布新版本时，需要先审阅 release notes、已知问题和升级风险，再执行固定升级流程。

**工作流形态：**

```text
每天 09:00
  -> 从白名单运行 service_version_check
  -> AI 总结当前版本、最新版本、release notes、已知问题和风险
  -> 创建手机审批请求
  -> 如果审批通过，入队 service_bundle_upgrade
  -> root runner 再次校验 action 名称，并执行固定脚本
```

**结果：** AI 负责阅读和风险摘要，人工决策作为执行边界。运行时不允许 AI 生成 shell 命令。

**示例文件：** `examples/service-upgrade-watch.yaml`

## 3. 多组件协同升级

有些系统需要同时评估并升级多个相关组件。

通用模式：

- 只读检查返回两个相关服务和一个管理组件的版本信息；
- AI 总结 release notes、公开 issue 反馈、兼容性风险和建议；
- 审批页提供两个稳定选项：**Upgrade** 和 **Skip**；
- 如果审批通过，一个白名单 action 将相关组件作为 bundle 升级。

部署专属值保留在公开仓库之外：

- 服务名和内部域名；
- 生产主机名和 SSH 细节；
- webhook URL 和 API token；
- 具体升级命令和回滚脚本。

## 4. 证书和域名过期提醒

**问题：** 证书或域名过期可能造成可避免的故障。

**工作流形态：**

```text
每天 08:00
  -> 运行只读过期检查
  -> 总结即将过期的资源
  -> 只在需要注意时通知
```

**变体：** 只有当固定续期或 DNS 动作通过白名单 action 接入时，才需要增加审批。

## 5. 备份健康报告

**问题：** 备份任务可能静默失败，直到需要恢复时才暴露问题。

**工作流形态：**

```text
每周一 08:00
  -> 运行只读备份状态检查
  -> AI 总结最近一次成功备份、缺失任务和风险
  -> 发送通知
```

**结果：** 运维人员收到的是运行摘要，而不是原始日志。

## 6. CI 失败归因

**问题：** CI 提醒噪音较大。有些失败是偶发的，有些需要快速处理。

**工作流形态：**

```text
工作时间每 30 分钟
  -> 抓取 CI 状态或运行只读 CLI 检查
  -> AI 按可能原因归类失败
  -> 发送最重要的处理项
```

**变体：** 可为固定安全动作增加审批按钮，例如重新运行某个已知 workflow。

## 7. 账单和订阅变化

**问题：** 云账单、SaaS 续费和用量突增容易被忽略。

**工作流形态：**

```text
每天早上
  -> 抓取或检查账单摘要
  -> AI 标出异常支出、即将续费项目和可能原因
  -> 发送通知
```

**安全提醒：** API 凭证应存放在部署专属配置中，而不是 workflow YAML。

## 8. 生活便利任务

同样模式也适合运维之外的场景：

- 关注商品的价格变化摘要；
- 出行前的航班、天气、交通风险摘要；
- 政策或签证更新的公开信息摘要；
- 每周阅读摘要；
- 家庭服务器磁盘、电池和备份健康检查。

适合的任务通常具备两个特征：重复、信息量大，并且偶尔需要简单 yes/no 决策。

## 怎么选择模式

| 需求 | 推荐模式 |
| --- | --- |
| 读取公开网页并总结 | `http_fetch` + `ai_summary` + `notify` |
| 检查内部系统状态 | `command_check` + `ai_summary` + `notify` |
| 执行前需要确认 | 加 `approval` |
| 审批后执行动作 | 加 `queued_action` 和部署方维护的白名单 |
| 需要任意 shell | 写固定脚本，并只白名单 action 名称 |
