# 使用场景

这些例子展示 `ai-approval-workflow` 最适合解决的问题：安静地定时自动化，只在需要你做决定时打扰你。

> 隐私说明：下面所有例子都使用通用服务名、占位路径和公开安全的描述。真实主机名、token、命令请放在你的私有 overlay 里。

## 1. 每日 GitHub Trending 摘要

**痛点：** 你想跟踪新库和新工具，但每天手动刷 trending 很浪费注意力。

**工作流形态：**

```text
每天 09:00
  -> 抓取 GitHub Trending
  -> AI 总结值得关注的仓库、语言分布和想法
  -> 发手机通知
```

**价值：** 你收到的是短摘要，不需要浏览长页面。这个任务只读，不需要审批。

**示例文件：** `examples/github-trending-daily.yaml`

## 2. 服务升级检查 + 手机审批

**痛点：** 服务有新版本时，需要有人看 release notes、已知问题和升级风险，再决定是否执行固定升级脚本。

**工作流形态：**

```text
每天 09:00
  -> 从白名单运行 service_version_check
  -> AI 总结当前版本、最新版本、release notes、已知问题和风险
  -> 发手机审批链接
  -> 如果审批通过，入队 service_bundle_upgrade
  -> root runner 再次校验 action 名称，并执行固定脚本
```

**价值：** AI 负责阅读和风险摘要，人负责最后决策。运行时永远不让 AI 临时生成 shell 命令。

**示例文件：** `examples/service-upgrade-watch.yaml`

## 3. 脱敏的真实升级场景

这个项目最初就是围绕一个真实运维需求设计的：检查一个生产业务系统及其相关管理组件是否值得升级。

公开安全版本：

- 工作流检查两个相关服务版本和一个管理组件版本；
- AI 阅读 release notes 和公开 issue 反馈；
- 手机审批提供两个按钮：**Upgrade** 和 **Skip**；
- 如果审批通过，一个白名单 action 会把相关组件作为一个 bundle 升级。

保持私有的内容：

- 真实服务名和域名；
- 生产主机名和 SSH 细节；
- webhook URL 和 API token；
- 具体升级命令和回滚脚本。

## 4. 证书和域名过期提醒

**痛点：** 证书或域名过期很容易漏掉，一旦影响生产就很麻烦。

**工作流形态：**

```text
每天 08:00
  -> 运行只读过期检查
  -> 总结即将过期的项目
  -> 只在需要注意时通知
```

**建议：** 普通提醒只通知即可。如果要接固定续期或 DNS 动作，再加审批。

## 5. 备份健康报告

**痛点：** 备份经常是静默失败，直到真正需要恢复时才发现。

**工作流形态：**

```text
每周一 08:00
  -> 运行只读备份状态检查
  -> AI 总结最近一次成功备份、缺失任务和风险
  -> 发手机通知
```

**价值：** 你看到的是运维摘要，而不是原始日志。

## 6. CI 失败归因

**痛点：** CI 提醒噪音大。有些失败是偶发的，有些需要尽快处理。

**工作流形态：**

```text
工作时间每 30 分钟
  -> 抓取 CI 状态或运行只读 CLI 检查
  -> AI 按可能原因归类失败
  -> 通知最重要的处理项
```

**可选审批：** 可以加一个固定安全动作，例如重新运行某个已知 workflow。

## 7. 账单和订阅变化

**痛点：** 云账单、SaaS 续费和用量突增容易被忽略。

**工作流形态：**

```text
每天早上
  -> 抓取或检查账单摘要
  -> AI 标出异常支出、即将续费项目和可能原因
  -> 发手机通知
```

**安全提醒：** API 凭证放私有配置，不要写进 workflow YAML。

## 8. 生活便利任务

同样模式也适合生活场景：

- 关注商品的降价摘要；
- 出行前的航班、天气、交通风险摘要；
- 政策或签证更新的公开信息摘要；
- 每周阅读摘要；
- 家庭服务器磁盘、电池和备份健康检查。

判断标准：如果一个任务重复、信息量大、偶尔需要简单 yes/no 决策，就很适合。

## 怎么选择模式

| 需求 | 推荐模式 |
| --- | --- |
| 读取公开网页并总结 | `http_fetch` + `ai_summary` + `notify` |
| 检查私有系统状态 | `command_check` + `ai_summary` + `notify` |
| 执行前需要确认 | 加 `approval` |
| 审批后执行动作 | 加 `queued_action` 和私有白名单 |
| 需要任意 shell | 不要这样做；写固定私有脚本，并只白名单脚本名称 |
